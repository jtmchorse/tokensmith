# tokensmith eval — does it actually make agents design-system-aware?

Tokensmith's claim is *"make coding agents design-system-aware."* This directory
turns that claim into a number, using the
[llm-eval-harness](https://github.com/jt-mchorse/llm-eval-harness).

It's a **two-arm eval** over the same golden set of UI micro-tasks:

| Arm | What the agent had | Mean on-system score |
|---|---|---|
| **control** | nothing — writes CSS from memory | **0.246** |
| **treatment** | tokensmith's resolved token table in context | **1.000** |
| **lift** | | **+0.754** |

10/12 rows improved, 0 regressed, **2 unchanged** — the honest ceiling: for
universal values (white text on a filled button, 16px body copy) the guess is
already right, so tokensmith adds nothing there. Its value concentrates on
*brand-specific* and *ambiguous-role* values, exactly where an agent's memory
drifts (`#2563eb` when the brand blue is `#2557c7`).

## Run it

```bash
python -m venv .venv && . .venv/bin/activate
pip install -r requirements.txt
python run_eval.py
```

No API key needed. The judge verdicts are recorded and replayed through the real
harness pipeline (`Judge → parse_judge_output → run_suite → diff_runs`), so the
numbers reproduce exactly and the run is deterministic.

## How it works

- **Golden set** — [`tokensmith_onsystem_v1.jsonl`](./tokensmith_onsystem_v1.jsonl):
  12 UI micro-tasks (8 color, 4 dimension), each with the correct semantic token
  + resolved value. Grounded in [`../examples/tokens.json`](../examples/tokens.json)
  (the "Meridian" demo design system) and verified against the live MCP
  (`resolve_token` / `list_tokens`). Passes the harness's own `validate` linter
  (12/12).
- **Two arms** — generated by independent subagents with **no leakage**: the
  control agent only saw the task text; the treatment agent also saw the resolved
  token table (what `list_tokens` returns).
- **Judge** — a human-in-the-loop scores on-system-ness on a fixed rubric:
  `1.0` if the literal equals the correct token's resolved value, `~0.1` for a
  near-miss (visually close but off-system drift), `0.0` for fully off. Verdicts
  live in [`run_eval.py`](./run_eval.py) and replay through the harness.
- **Collision rows** — `border.focus` shares `#2557c7` with `action.primary`;
  `text.on-action` shares `#ffffff` with `surface.default`. Treatment has to pick
  the right *name*, not just a matching value. Both passed.

## Recorded results

[`results/`](./results/) holds the last run's artifacts:
`control_run.json`, `treatment_run.json`, and `delta.json` (the per-row lift).

## Scope & limitations

- N=12, single non-blind judge — mitigated by an objective criterion (literal ==
  token value), but not a calibrated Cohen's-κ run.
- Treatment is *handed* the token table (simulating tokensmith availability); it
  does not test the agent **calling** `resolve_token` mid-generation — tokensmith's
  own stated open gap.
- Treatment scores a flat 1.0, so this v1 demonstrates the lift but does not yet
  discriminate treatment quality. A harder set (distractor tokens, ambiguous
  roles, self-service tool calls) is the natural v2.
